Theme 1 — Problem Framing

Why a Digital Twin? The 3 Operational Loops

Enterprise WiFi failures are hard to predict because RRM (AutoRF), client behaviour, and application load interact non-linearly. We framed the solution as three closed loops: Design Loop (pre-deploy prediction via link budget + airtime), Optimization Loop (live EP1 telemetry ↔ RRM simulation agent), and Troubleshooting Loop (event-driven 30-row agent). Each loop has its own agent, data sources, and feedback path into XIQ configuration.

Theme 2 — Scope Decision

Mixed-Mode Deferral: Why Phase 1 Is Uniform WiFi 6 or WiFi 7

The question: can the tool handle a site with both WiFi 6 and WiFi 7 APs? Yes, but only with a floor plan — because zone-boundary problems arise: MLO drops on inter-generation roam, 320 MHz→160 MHz ACI at borders, Multi-AP Coord clusters that only WiFi 7 APs join. Without an adjacency graph, these cannot be resolved. Decision: Phase 1 = one AP generation per site. User chooses WiFi 6 OR WiFi 7. Mixed-mode = Sprint 4+ with floor plan ingestion. The 12 additional DPM inputs needed are catalogued in LLD Appendix A.

Theme 3 — Code Architecture

Strategy Pattern: One Engine, Two Protocols

Rather than separate engines, we applied the Strategy Pattern. LinkBudgetStrategy ABC. WiFi6LinkBudget caps MCS at 11. WiFi7LinkBudget inherits WiFi 6 and upgrades to MCS 12/13 when SNR ≥ 36 dB + protocol == 802.11be. ComparisonEngine runs both on the same DPM and returns a delta report — so a single site survey instantly shows “WiFi 7 saves you 2 APs here.”

Theme 5 — Engine Calibration

Rev Wi-Fi Airtime Formula, CCI Scoring, and the VoIP Trap

Core formula: Airtime% = App_Kbps / Device_Kbps per active client. Available = RF_Environment% − SSID Beacon Overhead% (1 SSID = 2.5% on 5 GHz, 1.5% on 6 GHz; 6 SSIDs = 15%). AP count = ⌈Σ(Active × Airtime) / Available⌉. OFDMA modifier ×1.30 when >50% frames <256 bytes. VoIP trap: G.711 is 128 Kbps (not 64 Kbps) due to 160-byte frames, no AMPDU — airtime is 2.5× higher than 5G engineers expect. CCI: reuse = ap_count / channels; LOW < 1.5, MEDIUM < 3.0, HIGH ≥ 3.0.

Theme 6 — Agent Architecture

Intake → Design → Config: The Orchestrator State Machine

Intake Agent (L6): Socratic DPM dialogue, validates A–H. Design Agent (L7): iterative loop — link budget → airtime → AP count → channel plan → CCI check. If CCI = HIGH, back-propagates (reduce SSIDs or widen channel) and re-runs. Exits when CCI < 3.0 or max iterations. Config Agent (L8): emits XIQ YAML, EXOS CLI, VOSS CLI. Orchestrator: 12-state FSM (IDLE → INTAKE → DESIGN → CONFIG → DONE) with retry logic + WebSocket event bus for live React state.
