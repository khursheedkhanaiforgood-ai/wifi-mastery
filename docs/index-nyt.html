Design Journey — How We Designed This Architecture

A chronological account of the architectural decisions made on April 29, 2026 — from first principles to a fully-specified two-branch platform. Each step links to the document where that decision lives.

🌐
Theme 1 — Problem Framing

Why a Digital Twin? The 3 Operational Loops

Enterprise WiFi failures are hard to predict because RRM (AutoRF), client behaviour, and application load interact non-linearly. A reactive approach means catching problems after users complain. We framed the solution as three closed loops: Design Loop (pre-deploy prediction), Optimization Loop (live EP1 telemetry ↔ RRM simulation), and Troubleshooting Loop (event-driven 30-row agent). Each loop has its own agent, its own data sources, and its own feedback path back into XIQ configuration.

Theme 2 — Scope Decision

Mixed-Mode Deferral: Why Phase 1 Is Uniform WiFi 6 or WiFi 7

The user asked: can the tool handle a site with both WiFi 6 and WiFi 7 APs? The answer is technically yes, but only if we know the floor plan — because mixed-mode creates zone-boundary problems: MLO drops on inter-generation roam, 320 MHz→160 MHz ACI at borders, and Multi-AP Coordination clusters that only WiFi 7 APs can join. Without an adjacency graph, we cannot resolve those conflicts. Decision: Phase 1 = uniform AP generation per site. User chooses WiFi 6 OR WiFi 7. Mixed-mode is Sprint 4+ when floor plan ingestion is available. The 12 additional input parameters needed for mixed-mode are catalogued in LLD Appendix A.

Theme 3 — Code Architecture

Strategy Pattern: One Engine, Two Protocols

Rather than writing separate engines for WiFi 6 and WiFi 7, we applied the Gang-of-Four Strategy Pattern. LinkBudgetStrategy is an abstract base class. WiFi6LinkBudget caps MCS at 11. WiFi7LinkBudget inherits WiFi 6 and upgrades to MCS 12/13 only when SNR ≥ 36 dB and protocol == 802.11be. A ComparisonEngine runs both strategies on the same DPM and returns a delta report. This means a single site survey can instantly show: “WiFi 7 saves you 2 APs here because the open-plan office SNR consistently clears 36 dB.”

Theme 4 — Data Model

Design Parameter Manifest: 7 Sections, Socratic Intake, Source Flags

Every design decision is anchored to the DPM — a ~60-parameter manifest in 7 sections: A (Site), B (Clients), C (Applications), D (Network), E (Channel), F (Security), G (RRM), plus H (WiFi 7 extras: MLO bands, 320 MHz, Multi-AP Coord toggle). Each field carries a source flag: [User provided], [Derived from X], or [Default — confirm?]. The Intake Agent uses a Socratic dialogue to populate A–H before any engine runs. Pydantic v2 model validators enforce business rules (e.g. concurrent_active ≤ concurrent_assoc). RRMParams hard-codes dynamic_channel_width=False per WLANPros WLPC 2026 guidance.

Theme 5 — Engine Calibration

Rev Wi-Fi Airtime Formula, CCI Scoring, and the VoIP Trap

The capacity engine is built on the Revolution Wi-Fi Capacity Planner methodology. Core formula: Airtime% = App_Kbps / Device_Kbps per active client. Available airtime = RF_Environment% − SSID Beacon Overhead% (1 SSID = 2.5% on 2.4/5 GHz, 1.5% on 6 GHz; 6 SSIDs = 15.0%). AP count = ⌈Σ(Active × Airtime) / Available⌉. OFDMA modifier: ×1.30 when >50% frames <256 bytes. VoIP trap: G.711 is not 64 Kbps — it is 128 Kbps per stream (160-byte frames, no AMPDU), so airtime is 2.5× higher than engineers from 5G expect. CCI scoring: reuse ratio = ap_count / available_channels; LOW < 1.5, MEDIUM < 3.0, HIGH ≥ 3.0.

Theme 6 — Agent Architecture

Intake → Design → Config: The Orchestrator State Machine

Four agents form the Phase 1 pipeline. Intake Agent (L6) conducts the Socratic DPM dialogue and validates all 7 sections. Design Agent (L7) runs the iterative loop: link budget → airtime → AP count → channel plan → CCI check. If CCI = HIGH, it back-propagates by reducing SSIDs or increasing channel width and re-runs. The loop exits when CCI < 3.0 or max iterations reached. Config Agent (L8) emits XIQ template YAML, EXOS CLI commands, and VOSS CLI commands. Orchestrator is a 12-state FSM (IDLE → INTAKE → DESIGN → CONFIG → DONE) with error states, retry logic, and a WebSocket event bus so the React frontend can show live state transitions.

Theme 7 — Standards Basis

Three Authoritative Sources: IEEE 802.11be, WLANPros WLPC 2026, Rev Wi-Fi

Every parameter in the LLD is traceable to one of three sources. IEEE 802.11be-2024 (WiFi 7 spec): MCS table, MLO frame formats, EHT PPDU, 320 MHz channel bonding, Multi-AP Coordination TGbe draft. WLANPros WLPC 2026: 30-row troubleshooting cause table, 6-process framework (MANAGE / INVESTIGATE / MEASURE / VALIDATE / ANALYZE / INTERROGATE), and the definitive “never use dynamic channel width” guideline that is hard-coded into RRMParams. Revolution Wi-Fi Capacity Planner: the complete airtime formula derivation, SSID overhead calculator, Network Design input taxonomy, and RSSI tier definitions. The Teaching Agent bridges all three to 5G cellular equivalents throughout the HLD.
