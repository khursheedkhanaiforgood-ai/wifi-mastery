1. Vision

What this is: A true WiFi Digital Twin — virtual representation of an enterprise WiFi network that predicts outcomes before hardware is touched, mirrors the live network via EP1 telemetry, runs the same RRM algorithm Broadcom chipsets run, and continuously tunes XIQ configuration based on the gap between predicted and actual behavior.
🔮 Predictive (Pre-Deploy)
  • Capacity plan (airtime demand model)
  • Link budget per device type
  • Channel plan (CCI scoring)
  • AP count = max(coverage, capacity)
  • XIQ config + EXOS/VOSS CLI output
🔄 Live Mirror (Post-Deploy)
  • EP1 telemetry ingested every 15 min
  • RRM Sim mirrors Broadcom AutoRF
  • Predicted vs actual gap analysis
  • Config delta → XIQ API push
  • Coverage/capacity balance maintained
🎓 Teaching Throughout
  • 5G↔WiFi bridge at every step
  • 802.11be standard citations
  • Socratic learning mode
  • WLANPros + WLPC 2026 corpus
  • Private RAG, never on GitHub

2. The Three Operational Loops

⚙️ Loop 1 — Design
Trigger: new site or major change
  1. Requirements intake (Socratic)
  2. Capacity planning (airtime model)
  3. Link budget (SNR→MCS→rate)
  4. AP selection (PoE + chipset match)
  5. Channel planning (CCI scoring)
  6. Config generation (XIQ + EXOS + VOSS)
  7. Deploy + qualify (WLANPros checklist)
🔄 Loop 2 — Optimization
Trigger: continuous (every 15 min)
  1. EP1 telemetry ingested
  2. RRM Sim runs (DCS + TPC + BSS Color)
  3. Predicted vs actual comparison
  4. Coverage/capacity gap detected
  5. Broadcom param identified
  6. Config delta generated
  7. XIQ API push + validate
🔍 Loop 3 — Troubleshoot
Trigger: EP1 alert or user symptom
  1. EP1 alert or symptom intake
  2. 30-row diagnosis (L1→L7)
  3. Spectrum analysis (13 signatures)
  4. Debug commands (EXOS/VOSS/XIQ)
  5. WLANPi guided testing
  6. Fix recommendation
  7. Config delta + push if applicable

4. RRM Dynamic Layer — The Twin Core

Why this is the most important layer: In 5G, SON runs centrally with full network visibility. In WiFi, every AP runs RRM independently via CSMA/CA — emergent behavior that is hard to predict. The RRM Simulation Agent runs the same Broadcom AutoRF algorithm in software, predicting what real APs will do BEFORE they do it.
DCS Simulator

Replicates Broadcom Dynamic Channel Selection algorithm.

Inputs: neighbor RSSI scan, channel utilization per channel, CCI threshold, retry rate trigger.

Output: predicted channel assignment per AP after AutoRF convergence. Detects oscillation risk.

≈ SON ANR + CCO in 5G
TPC Simulator

Replicates Broadcom TX Power Control step-down algorithm.

Inputs: neighbor AP signal levels, WLANPros targets (7/13/16 dBm), coverage RSSI floor.

Output: predicted TX power per radio. Detects coverage holes from over-reduction.

≈ SON CCO power component
BSS Color Simulator

802.11ax BSS Color assignment and conflict detection.

Inputs: neighbor AP scan, current color assignments (1-63).

Output: color conflict map + re-assignment recommendations. Reduces unnecessary deferrals between non-overlapping BSSs.

≈ ICIC in LTE/NR
Multi-AP Coord Sim

WiFi 7 (AP5020/5022/5060 only). Coordinated null-steering + spatial reuse scheduling.

Inputs: AP5020 neighbor topology, per-client SNR measurements.

Output: coordinated transmission schedule across multiple APs.

≈ CoMP in NR

5. Coverage vs Capacity — The Balance

The fundamental tension: More TX power = better coverage = more CCI = less capacity. Narrower channel width = less CCI = more capacity = less per-link throughput. Every RRM parameter is a trade-off knob on this spectrum. The Digital Twin manages both constraints simultaneously.
📡 Coverage-Driven AP Count
  • RF Coverage Design RSSI tier (−67/−75/−80 dBm)
  • Link Budget: SNR → MCS → min acceptable rate
  • Cell radius from path loss model
  • AP placement density from cell radius
  • Input: site dimensions + environment type
  • Tool: RF survey (Ekahau/Hamina)
  • Constraint: every point must reach RSSI target
= max ✦
⚡ Capacity-Driven AP Count
  • Airtime demand per device type
  • Active device count × airtime per device
  • Available airtime − SSID beacon overhead
  • AP radio count to serve total active airtime
  • Input: device mix + application SLAs
  • Tool: Revolution Wi-Fi Capacity Planner
  • Constraint: channel utilization <20% (5GHz)

Coverage/Capacity Tunable Levers

ParameterIncrease CoverageIncrease CapacityTrade-off
TX Power ↑Larger cell radiusMore CCI from neighborsWLANPros: 7/13/16 dBm target
Channel Width widerHigher per-link rateFewer non-overlapping channelsWider until CCI appears
MBR higher (12→24 Mbps)Smaller cell (weak clients excluded)Less airtime per beaconStart 12, try 24 in high-density
RX-SOP raise thresholdIgnores weak neighbor signalsAllows more parallel transmissionsHigh-density only, use with caution
Association Limit ↓No effectPrevents airtime starvationUse when capacity bottleneck is association
Add more APsFill coverage holesMore non-overlapping channelsOnly if CCI budget allows
Iterative Loop: Design Agent runs Capacity Plan → Coverage Plan → if gap >20%, adjusts channel width / TX power / MBR → re-runs both → iterates until both constraints satisfied. Post-deploy, Tuning Agent continuously re-solves this optimization using live EP1 data.

6. Extreme Platform One (EP1) Integration

EP1 is the observability and AI operations layer. For the Digital Twin, EP1 is the measurement plane — the source of ground truth about what the deployed network is actually doing.

EP1 → Digital Twin (Observe)
  • Channel utilization per radio → RRM Sim
  • RSSI per client → Link Budget validation
  • SNR per client → MCS validation
  • Retry rate per radio → DCS trigger input
  • Roaming events → 802.11r/k/v validation
  • Client MCS distribution → capacity validation
  • BSS Load IE → load vs design comparison
  • AP health / reboots → hardware troubleshooting
  • BSS color conflicts → RRM sim input
  • OFDMA RU utilization → efficiency measurement
  • TWT session stats → battery/IoT validation
Digital Twin → EP1/XIQ (Control)
  • Config delta → XIQ Device Template update
  • Channel reassignment → XIQ AutoRF override
  • TX power adjustment → XIQ TPC settings
  • MBR change → XIQ radio profile update
  • RX-SOP tuning → XIQ high-density profile
  • SSID reduction → XIQ template edit
  • Troubleshoot fix → EXOS/VOSS CLI push
  • All changes: logged + teaching commentary

EP1 Feedback Loop Example

EP1 reports: AP-07, radio 5GHz, channel utilization 85%, retry rate 28%
    │
    ▼ EP1 Integration Agent normalizes + stores time-series
    │
    ▼ RRM Simulation Agent
      Runs DCS algorithm: 85% util → triggers channel change
      Predicts: AutoRF will move AP-07 from ch 36 → ch 100
      BUT: ch 100 has 3 neighbor APs at -80dBm → will cause CCI
      Result: oscillation risk DETECTED (RED)
    │
    ▼ Tuning Agent
      Root cause: 6 SSIDs broadcasting → 22% beacon overhead
                  Available airtime only 56% (should be 78%)
      Fix: reduce to 4 SSIDs on this radio + increase CCI threshold to -80dBm
    │
    ▼ Config delta generated:
      XIQ template: disable 2 SSIDs on AP-07 5GHz radio
      XIQ template: update CCI threshold -85→-80 dBm
    │
    ▼ Teaching Agent:
      "This mirrors PRB overhead in 5G. Your 6 SSIDs consume 22% of your
       'radio frame'. Reducing to 4 SSIDs is equivalent to reducing PDCCH
       + pilot overhead — more PRBs available for data. Same airtime math."
    │
    ▼ XIQ API push → EP1 confirms: utilization drops to 61%, retries to 8%

Critical insight for 5G engineers: Only 25% of WiFi frames carry user data (vs >90% in 5G). 80% of all WiFi frames are <256 bytes. 60% of all frames are control frames. This is why OFDMA is so impactful — and why airtime, not bandwidth, is the correct capacity metric.

9. WiFi Generation Strategy

Version 1 scope: User selects AP generation — WiFi 6 (AP3000/AP4000) or WiFi 7 (AP5020/AP5022/AP5060) — for the entire site. The tool designs uniformly for that generation. The Comparison Engine runs both strategies on the same DPM inputs and shows the delta. Mixed-mode (WiFi 6 + WiFi 7 on the same site) is deferred to a future sprint — it requires floor plan topology which Version 1 does not collect.

Two Branches — Same Engine, Different Strategy

feature/wifi6-baseline
AP fleet: AP3000 (2×2:2) / AP4000 (tri-radio 6E)
Max MCS: 11 (1024-QAM, SNR ≥33 dB)

feature/wifi7-complete

Strategy Pattern — How the Engine Implements This

class LinkBudgetStrategy(ABC):
    @abstractmethod
    def snr_to_mcs(self, protocol, channel_width, snr_db, spatial_streams) → MCSResult: ...
    @abstractmethod
    def max_mcs_index(self) → int: ...

class WiFi6LinkBudget(LinkBudgetStrategy):
    def max_mcs_index(self) → int: return 11        # MCS 11 = 1024-QAM 5/6

class WiFi7LinkBudget(WiFi6LinkBudget):             # inherits WiFi 6 baseline
    def snr_to_mcs(...):
        result = super().snr_to_mcs(...)             # WiFi 6 lookup first
        if snr_db >= 36 and self.be_capable:
            result = self._apply_mcs_12_13(snr_db)  # upgrade to 4K-QAM
        return result
    def max_mcs_index(self) → int: return 13        # MCS 13 = 4K-QAM 5/6

# Comparison Engine — runs BOTH strategies on the SAME DPM
class ComparisonEngine:
    def run(self, dpm: DesignParameterManifest) → ComparisonReport:
        wifi6  = DesignEngine(strategy=WiFi6Strategy()).run(dpm)
        wifi7  = DesignEngine(strategy=WiFi7Strategy()).run(dpm)
        return ComparisonReport(wifi6=wifi6, wifi7=wifi7, delta=self._diff(wifi6, wifi7))

Mixed-Mode — Deferred (future sprint)

Why deferred: Mixing WiFi 6 APs (AP3000/AP4000) and WiFi 7 APs (AP5020+) on the same site introduces zone-boundary problems — MLO loss on roam between generations, 320→160 MHz ACI at adjacency, Multi-AP Coord cluster limited to WiFi 7 APs only. Solving these correctly requires knowing which AP is physically adjacent to which. That requires floor plan input, AP placement, and an adjacency graph — none of which Version 1 collects.

    11. Why This Is a True Digital Twin

    Digital Twin CriterionThis Platform
    Virtual model of physical system✅ Agent-simulated WiFi: capacity, link budget, RRM, channel
    Real-time data from physical system✅ EP1 telemetry ingested every 15 min
    Bidirectional synchronization✅ EP1 → Twin (observe) + Twin → XIQ (control)
    Predictive simulation✅ RRM Sim predicts AutoRF convergence before it happens
    Optimization feedback loop✅ Gap analysis → config delta → push → EP1 validates
    Mirroring of physical behavior✅ Broadcom AutoRF algorithm replicated: DCS + TPC + BSS Color
    Multi-domain coverage✅ RF layer, capacity layer, config layer, wired layer
    The core insight: The physical network and the digital twin run the same RRM algorithm — one in Broadcom silicon, one in Python. When they diverge, the Tuning Agent finds out why and corrects it. This is what makes it a real twin, not just a design tool.
