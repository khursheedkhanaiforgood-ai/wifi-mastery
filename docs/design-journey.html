Theme 1 — Problem Framing

Why a Digital Twin? The 3 Operational Loops

Enterprise WiFi failures are hard to predict because RRM (AutoRF), client behaviour, and application load interact non-linearly. A reactive approach means catching problems only after users complain. The question was: what is the minimum viable architecture that closes the loop between prediction and production?

We framed the solution around three closed feedback loops:

Design Loop — pre-deployment prediction using link budget, airtime formula, and channel planning. No physical hardware required. Output: AP count, channel plan, XIQ config template.

Optimization Loop — live EP1 telemetry feeds an RRM Simulation Agent that predicts the outcome of each AutoRF adjustment before committing it. Mirrors Broadcom's DCS/TPC cycle with a 30-minute convergence window.

Troubleshooting Loop — event-driven 30-row agent triggered by EP1 anomaly events. Classifies root cause (WIRELESS / LOCAL NET / INTERNET) and recommends remediation steps.

Decision: Each loop has its own agent, its own data sources, and its own feedback path into XIQ. They share the DPM as a common data schema but never cross-call at runtime.

Theme 2 — Scope Decision

Mixed-Mode Deferral: Why Phase 1 Is Uniform WiFi 6 or WiFi 7

The question: can the tool handle a site with both WiFi 6 and WiFi 7 APs simultaneously? Technically yes, but only if we know the floor plan — because mixed-mode creates zone-boundary problems that cannot be resolved without topology:

MLO drops on inter-generation roam — a WiFi 7 client's MLO session cannot continue when it roams to a WiFi 6 AP. The Design Agent cannot predict this without knowing which APs are adjacent.

320 MHz → 160 MHz ACI at zone borders — a WiFi 7 AP running 320 MHz creates adjacent-channel interference for WiFi 6 APs running 160 MHz in adjacent zones. Severity depends on physical distance.

Multi-AP Coordination cluster = WiFi 7 only — the coordination cluster cannot include WiFi 6 APs. Without a floor plan the cluster boundaries are unknown.

Decision: Phase 1 = uniform AP generation per site. The user chooses WiFi 6 OR WiFi 7. Mixed-mode is explicitly deferred to Sprint 4+ when floor plan ingestion is available. The 12 additional DPM input parameters needed for mixed-mode are catalogued in LLD Appendix A so they are not lost.

Theme 3 — Code Architecture

Strategy Pattern: One Engine, Two Protocols

Phase 1 must handle two deployment scenarios (WiFi 6 and WiFi 7) using the same DPM inputs, the same engines, and the same orchestrator. The naive approach — two separate engines — would mean duplicating the entire link budget, airtime, and channel planning logic with only the MCS table differing.

We applied the Gang-of-Four Strategy Pattern: a single abstract base class with two concrete strategies differing only in MCS ceiling and SNR thresholds.

LinkBudgetStrategy (ABC) → WiFi6LinkBudget caps MCS at 11. WiFi7LinkBudget inherits WiFi 6 and overrides: if SNR ≥ 36 dB AND protocol == 802.11be, use MCS 12/13. Otherwise fall through to parent. A ComparisonEngine runs both strategies on the same DPM and returns a ComparisonReport with per-AP delta.

Decision: Strategy Pattern. WiFi7LinkBudget inherits WiFi6LinkBudget (not a sibling). This means WiFi 7 gracefully falls back to MCS 11 behaviour in poor SNR environments — the same code path, no separate branch logic.

Theme 4 — Data Model

Design Parameter Manifest: 7 Sections + Socratic Intake + Source Flags

Every engine must anchor to a common schema so that the Intake Agent, Design Agent, Config Agent, and Orchestrator speak the same language. Ad-hoc dictionaries would make validation brittle and make the comparison report meaningless.

The Design Parameter Manifest (DPM) is ~60 parameters in 7 sections: A (Site), B (Clients), C (Applications), D (Network), E (Channel), F (Security), G (RRM). WiFi 7 deployments extend with H (MLO bands, 320 MHz toggle, Multi-AP Coord on/off).

Each field carries one of three source flags: [User provided], [Derived from X], or [Default — confirm?]. This allows the Design Agent to surface uncertainty rather than silently applying defaults.

Pydantic v2 model validators enforce cross-field business rules. Example: concurrent_active_pct ≤ concurrent_assoc_pct. RRMParams.dynamic_channel_width is hard-coded False per WLANPros WLPC 2026 guidance — it is never surfaced as a user input.

Decision: Pydantic v2 with @model_validator for cross-field constraints. Source flags are stored as metadata alongside each value, not separate fields. The Intake Agent uses a Socratic dialogue to populate sections A–H before any engine is called.

Theme 5 — Engine Calibration

Rev Wi-Fi Airtime Formula, CCI Scoring, and the VoIP Trap

The capacity engine is built on the Revolution Wi-Fi Capacity Planner methodology. The core formula is deceptively simple but contains several non-obvious traps for engineers coming from the 5G domain.

Core formula: Airtime% = App_Kbps / Device_Kbps per active client. AP count = ⌈Σ(Active × Airtime) / Available⌉ where Available = RF_Environment% − SSID Beacon Overhead%.

SSID overhead: 1 SSID = 2.5% on 2.4/5 GHz, 1.5% on 6 GHz. 6 SSIDs = 15.0% on 2.4/5 GHz — a significant chunk of available airtime gone before any client transmits.

VoIP trap: G.711 is not 64 Kbps — it is 128 Kbps per stream because of 160-byte frames with no AMPDU aggregation. This makes VoIP airtime 2.5× higher than a 5G engineer would expect from PRB utilization analogy.

OFDMA modifier: ×1.30 efficiency gain when more than 50% of active devices have frames smaller than 256 bytes. Applies to IoT-dense and chat-heavy deployments.

CCI scoring: reuse ratio = ap_count / available_channels. LOW < 1.5, MEDIUM < 3.0, HIGH ≥ 3.0. The Design Agent back-propagates when CCI = HIGH.

Decision: VoIP is treated as a special-case application in the App model with override_frame_size=160 and ampdu=False. The OFDMA modifier is applied automatically based on the app profile mix — not a user toggle.

Theme 6 — Agent Architecture

Intake → Design → Config: The Orchestrator State Machine

Four agents form the Phase 1 pipeline. The key design question was: how tightly should they couple? A monolithic pipeline would be simple to implement but impossible to test at the module level. Fully async microservices would over-engineer Sprint 1.

Intake Agent (L6): Socratic DPM dialogue, validates sections A–H, emits a complete DPM object. Stateless between sessions (DPM persisted by Orchestrator).

Design Agent (L7): iterative capacity↔RF loop. link budget → airtime → AP count → channel plan → CCI check. If CCI = HIGH, back-propagates: reduces SSID count OR increases channel width and re-runs. Loop exits when CCI < 3.0 OR max_iterations (default 5) reached.

Config Agent (L8): stateless transformer. Takes a DesignResult and emits XIQ template YAML, EXOS CLI command block, VOSS CLI command block. No network calls.

Orchestrator: 12-state FSM — IDLE → INTAKE → INTAKE_RETRY → DESIGN → DESIGN_RETRY → CHANNEL → CONFIG → VALIDATE → DONE / ERROR_INTAKE / ERROR_DESIGN / ERROR_CONFIG. WebSocket event bus emits state transitions in real time.

Decision: Agents are synchronous Python classes in Sprint 1 (no async). The Orchestrator wraps them in an async FastAPI endpoint so the frontend sees async behaviour via WebSocket without requiring async internals in Sprint 1.

Theme 7 — Standards Basis

Three Authoritative Sources: IEEE 802.11be, WLANPros WLPC 2026, Rev Wi-Fi

Every parameter in the LLD is traceable to one of three authoritative sources. This was a deliberate design choice: parameters not traceable to one of these three require explicit justification in the code comment.

IEEE 802.11be-2024 (WiFi 7 spec): MCS 0–13 table, MLO frame formats (EMLSR/STR), EHT PPDU structure, 320 MHz channel bonding rules, Multi-AP Coordination TGbe draft, Preamble Puncturing, HARQ retransmission.

WLANPros WLPC 2026: 30-row potential causes table (WIRELESS / LOCAL NET / INTERNET), 6-process troubleshooting framework (MANAGE / INVESTIGATE / MEASURE / VALIDATE / ANALYZE / INTERROGATE), frame statistics interpretation, and the definitive guidance: never enable dynamic channel width — hard-coded as RRMParams.dynamic_channel_width = False with a code comment citing WLPC 2026.

Revolution Wi-Fi Capacity Planner (pp.1–38): complete airtime formula derivation, SSID overhead calculator (all frequency bands), Network Design input taxonomy (all DPM Section C fields), RF Coverage RSSI tier definitions, Capacity Results output schema.

Decision: The Teaching Agent (embedded in the Design Agent) maps every WiFi concept to its 5G/cellular equivalent before delivering it to the user. This is not a documentation feature — it is a runtime behaviour so 5G engineers receive explanations calibrated to their existing mental model.
